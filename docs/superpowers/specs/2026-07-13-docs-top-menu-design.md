# 文档顶部菜单选中态修复设计

## 背景

文档站顶部导航目前通过手写 `RouterLink` 和路径前缀判断维护选中态。“组件”菜单项的入口是 `/components/point-layer`，因此用户从左侧导航进入其他 `/components/*` 页面后，顶部路径判断不再匹配，“组件”的选中效果会消失。

## 目标

- 顶部导航改用 Element Plus 的横向 `el-menu`。
- 访问任意 `/components/*` 页面时，顶部“组件”菜单始终保持选中。
- 点击顶部菜单仍跳转到现有入口页：“指南”进入 `/`，“组件”进入 `/components/point-layer`。
- 左侧导航、路由配置和现有导航数据结构保持不变。
- 浅色与深色主题下均保持现有视觉层级，窄屏布局不退化。

## 方案

在 `DocsLayout.vue` 中新增由当前路由计算的顶部菜单分类键：根路径及指南页面映射为 `/`，所有 `/components/*` 页面映射为 `/components`。顶部 `el-menu` 使用分类键作为 `default-active`，菜单项点击事件再将分类键映射到 `topNavItems` 中已有的实际入口路径并交给 Vue Router 跳转。

这种设计将“当前所属分类”和“分类入口地址”分开：左侧导航只负责具体文档页跳转，顶部菜单只显示一级分类状态，因此两者不会再争用或覆盖同一个选中状态。

## 组件与数据流

1. Vue Router 更新 `route.path`。
2. `activeTopMenu` 根据路由路径返回 `/` 或 `/components`。
3. 横向 `el-menu` 接收该值并显示唯一的一级激活项。
4. 用户点击顶部菜单时，选择处理函数查找对应入口并调用 `router.push()`。
5. 左侧 `RouterLink` 继续按现有规则显示父级与子级激活状态。

不存在匹配分类时回退到 `/`，避免菜单处于无选中态。重复选择当前入口时交由 Vue Router 正常去重，不增加额外错误提示。

## 样式

移除仅服务于手写顶部链接的激活样式，改为限定在 `.docs-header__nav` 下的 Element Plus 菜单样式。背景、文字、悬停与激活颜色继续引用 `website/src/assets/styles/index.scss` 已有语义化主题变量，不新增只适用于单一主题的硬编码颜色；保留当前头部高度、间距和窄屏行为。

## 测试与验证

- 新增文档站导航回归测试，先验证当前实现缺少 `el-menu` 与统一的 `/components` 分类激活映射。
- 实现后验证顶部使用横向 `el-menu`，激活值来自当前路由分类，点击菜单通过路由跳转。
- 验证任意 `/components/*` 路径都映射为同一个顶部“组件”选中项。
- 运行相关 Vitest、完整 `npm test` 与 `npm run docs:build`。
- 在浅色、深色及窄屏布局下检查顶部菜单的选中、悬停和排版效果。

## 非目标

- 不把左侧导航改造成 `el-menu`。
- 不调整文档信息架构、路由地址或菜单文案。
- 不修改地图能力库的公开 API 与构建产物。

## 视觉细化

顶部菜单保持 Element Plus 横向菜单结构，但不再使用块状悬停背景或圆角菜单项。普通状态只显示中性文字；悬停时仅改变文字颜色；当前一级分类使用主题蓝色文字和 Element Plus 原生 2px 底部边框表示选中状态。

实现只调整 `website/src/assets/styles/index.scss` 中顶部菜单的局部变量与菜单项规则：将 `--el-menu-hover-bg-color` 设为透明，移除菜单项 `border-radius`，并继续使用 `--el-menu-active-color: var(--doc-primary-deep)`。不增加额外 `<span>` 或伪元素，因此下划线沿菜单项内容盒宽度显示，浅色和深色主题均复用现有语义变量。

回归测试需明确禁止非透明悬停背景和菜单项圆角，并保留激活色及默认底部边框。浏览器检查覆盖浅色、深色与窄屏，确认顶部菜单、版本徽标、主题按钮和 GitHub 链接无重叠。
